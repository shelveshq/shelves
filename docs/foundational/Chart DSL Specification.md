# DSL Specification: Chart Grammar

This document defines the Chart DSL — the domain-specific language for defining individual visualizations. It is written in YAML, designed to be authored directly by BI analysts or generated by an LLM from natural language prompts. It translates deterministically to Vega-Lite JSON.

The companion **Layout DSL** for composing dashboards is documented separately in `Layout_DSL_Specification.md`.

---

## Part 1: Chart DSL

### 1.1 Design Philosophy

The Chart DSL uses Tableau's shelf mental model as its vocabulary. An analyst who thinks in terms of rows, columns, marks, color, and detail can write a chart spec without learning Vega-Lite. The DSL acts as a constrained intermediate representation — compact enough for an LLM to produce reliably, expressive enough to cover the most common BI chart patterns.

**Principles:**
- Vocabulary mirrors Tableau: `rows`, `cols`, `marks`, `color`, `detail`, `filters`, `tooltip`, `sort`
- **`rows`/`cols` are exclusively for axis encoding; `facet` is a separate property for chart partitioning.** This is a deliberate improvement over Tableau, where the Rows/Columns shelves conflate axis assignment with small-multiple generation. In this DSL, the intent is always unambiguous.
- References semantic layer metrics/dimensions by name (never raw SQL or column expressions)
- Styling is excluded — the theme layer handles all visual aesthetics
- Each YAML file defines exactly one chart ("sheet" in Tableau terms)

### 1.2 Grammar Reference

#### Top-Level Properties

```yaml
# chart_name.yaml
sheet: "Revenue by Country"           # Display name
description: "Weekly revenue bars..."  # Optional description

# Data source — references a semantic layer model
data:
  model: "orders"                      # Semantic layer model/cube name
  measures: [revenue, arpu]            # Metrics to query
  dimensions: [country, week]          # Dimensions to include

# Shelf assignments (axis encoding only — never used for partitioning)
cols: <shelf_definition>               # X-axis / horizontal encoding
rows: <shelf_definition>               # Y-axis / vertical encoding

# Mark definitions
marks: <mark_definition>               # Default mark type (inherited by all measure entries)

# Encoding channels (top-level = defaults, inherited by measure entries/layers)
color: <encoding_definition>           # Color mapping
detail: <encoding_definition>          # Disaggregation without visual encoding
size: <encoding_definition>            # Size mapping
tooltip: <tooltip_definition>          # Tooltip fields

# Interactions
filters: <filter_definition>           # Shelf filters (hardcoded in spec)
sort: <sort_definition>                # Sort order

# Partitioning (small multiples — separate from axis encoding)
facet: <facet_definition>              # Row, column, or wrapping grid partitioning
```

### 1.3 Property Definitions

#### `data` — Data Source

```yaml
data:
  model: "orders"                   # Required: semantic layer model name
  measures:                         # Required: list of metrics to query
    - revenue
    - arpu
  dimensions:                       # Required: list of dimensions
    - country
    - week
  time_grain:                       # Optional: time dimension granularity
    field: week
    grain: week                     # day | week | month | quarter | year
```

The translator uses `model`, `measures`, and `dimensions` to construct the semantic layer API call. The returned data is bound to the Vega-Lite spec's `data` property.

#### `cols` and `rows` — Axis Encoding (Not Partitioning)

`cols` defines what goes on the x-axis. `rows` defines what goes on the y-axis. These are strictly for axis encoding — they never partition the chart into small multiples. Use `facet` for partitioning.

A shelf is either a **single field name** (string) or a **list of measure entries** (for multi-measure charts). At most one of rows/cols can be a list — enforced by a schema validator.

```yaml
# Shape 1: Single field (string) — simple chart
cols: week
rows: revenue

# Shape 2: List of measure entries — multi-measure (stacked panels or layers)
cols: week
rows:
  - measure: revenue
    mark: bar
    color: country
  - measure: orders
    mark: line
```

Each measure entry is a `MeasureEntry` object:

```yaml
- measure: revenue               # Required: field name from semantic layer
  mark: bar                      # Optional: override top-level marks
  color: country                 # Optional: override top-level color
  detail: country                # Optional: disaggregation field
  size: 200                      # Optional: size mapping or fixed value
  opacity: 0.8                   # Optional: mark opacity (0–1)
  axis: independent              # Optional: scale resolution for layers
  layer:                         # Optional: additional measures overlaid (Phase 1a)
    - measure: arpu
      mark:
        type: line
        style: dashed
      color: "#666666"
```

**Translation rules:**
- Single field → `encoding.x` or `encoding.y` with auto-detected type
- Single-item list with no layers → treated same as single field
- Multi-item list, no layers → `repeat` (same marks) or `vconcat` (mixed marks)
- Entry with `layer` → `layer` array with multiple marks
- Mixed entries (some with layers, some without) → `vconcat` of layer + simple specs

**Encoding inheritance:** Properties cascade top-level → measure entry → layer entry. More specific wins. See the Inheritance section below.

**What NOT to do:** Do not put multiple dimensions on `rows` or `cols` to create nested headers or small multiples. Use the `facet` property instead. This keeps the DSL unambiguous — `rows`/`cols` always mean axis encoding.

#### `marks` — Mark Definitions

The top-level `marks` property sets the default mark for all measure entries that don't specify their own.

```yaml
# Single mark (string shorthand)
marks: bar

# Single mark with properties
marks:
  type: line
  style: dashed                    # solid (default) | dashed | dotted

# Mark with additional properties
marks:
  type: bar
  opacity: 0.8
  point: true                    # Show data points on line marks
```

Mark types override at the measure entry or layer level:

```yaml
marks: line                        # Top-level default: all entries get line

rows:
  - measure: revenue
    mark: bar                      # Override: bars for this entry
  - measure: orders                # Inherits: line from top level
  - measure: arpu
    mark:
      type: line
      style: dashed                # Override with properties
```

**Supported mark types:**
| Mark | Vega-Lite `mark.type` | Notes |
|---|---|---|
| `bar` | `bar` | |
| `line` | `line` | `style: dashed` → `strokeDash: [6, 4]` |
| `area` | `area` | |
| `circle` | `circle` | For scatter plots |
| `square` | `square` | |
| `text` | `text` | Typically layered on another mark |
| `point` | `point` | Generic point mark |
| `rule` | `rule` | Reference lines |
| `tick` | `tick` | |
| `arc` | `arc` | Pie / donut charts |
| `rect` | `rect` | Heatmaps |
| `geoshape` | `geoshape` | Maps |

#### `color` — Color Encoding

Color can be a field mapping or a fixed value. At the top level, it applies as a default to all measure entries.

```yaml
# Map a dimension to color (categorical)
color: country

# Map a measure to color (sequential/gradient)
color:
  field: revenue
  type: quantitative

# Fixed color value (overrides theme)
color: "#4A90D9"
```

Per-entry override:

```yaml
color: country                       # Top-level default

rows:
  - measure: revenue
    mark: bar
    # color: country               # Inherited from top-level
  - measure: orders
    mark: line
    color: "#E5A84B"               # Override: fixed gold for this entry
```

Layer-level override:

```yaml
rows:
  - measure: revenue
    mark: bar
    color: country                   # Primary mark colored by country
    layer:
      - measure: arpu
        mark: line
        color: "#666666"             # Layer: fixed grey, overrides entry color
```

**Translation rules:**
- Field name → `encoding.color: { field: "country", type: "nominal" }`
- Hex value → `mark.color: "#4A90D9"` or `encoding.color: { value: "#666666" }`
- Per-entry/layer values override the parent level's color

#### `detail` — Detail / Level of Disaggregation

Detail breaks marks into groups without applying a visual encoding. This is identical to Tableau's Detail shelf.

```yaml
# Single field
detail: country

# Explicitly null in a layer entry — aggregates across that dimension
layer:
  - measure: avg_revenue
    mark: rule
    detail: null                     # Do NOT inherit parent detail
```

**Translation:** `detail: country` → `encoding.detail: { field: "country", type: "nominal" }`

A `null` value means the mark aggregates across all values of that dimension — critical for patterns like "bars by country + overall average line."

#### `size` — Size Encoding

```yaml
# Map field to size
size: revenue

# Fixed size
size: 200
```

#### `tooltip` — Tooltip Fields

```yaml
# Simple list
tooltip: [country, revenue, arpu]

# With formatting
tooltip:
  - field: country
  - field: revenue
    format: "$,.0f"
  - field: arpu
    format: "$,.2f"
```

#### `filters` — Shelf Filters (Hardcoded)

These are data filters baked into the chart spec, not interactive filter panes (those live in the layout DSL).

```yaml
filters:
  - field: country
    operator: in                   # in | not_in | eq | neq | gt | lt | gte | lte | between
    values: ["US", "UK", "FR"]

  - field: revenue
    operator: gt
    value: 1000

  - field: date
    operator: between
    range: ["2024-01-01", "2024-12-31"]
```

**Translation:** Each filter becomes a `transform: [{ filter: ... }]` entry in the Vega-Lite spec.

#### `sort` — Sort Order

```yaml
# Sort by field value (ascending)
sort:
  field: revenue
  order: descending                # ascending | descending

# Explicit custom order
sort:
  field: country
  order: ["US", "UK", "FR", "DE", "JP"]

# Sort by encoding (e.g., sort x-axis bars by y value)
sort:
  encoding: y
  order: descending
```

#### `facet` — Chart Partitioning (Small Multiples)

Facet is the sole mechanism for partitioning a chart into small multiples. It is deliberately separate from `rows`/`cols` to maintain a clean distinction: `rows`/`cols` control what goes on each axis, `facet` controls how the chart is split into panels.

```yaml
# Facet into rows (vertical stack of panels)
facet:
  row: region

# Facet into columns (horizontal row of panels)
facet:
  column: product_category

# Facet into both rows and columns (grid)
facet:
  row: region
  column: product_category

# Wrapping facet (single dimension, wraps into a grid)
facet:
  field: country
  columns: 4                       # Wrap after 4 columns
  sort: ascending                  # Optional sort order for panels

# Facet with independent scales (each panel gets its own axis range)
facet:
  row: region
  resolve:
    scale:
      y: independent               # independent | shared (default: shared)
```

**Translation rules:**
- `facet.row` → Vega-Lite `row` encoding channel or `facet` operator with `row`
- `facet.column` → Vega-Lite `column` encoding channel or `facet` operator with `column`
- `facet.field` + `facet.columns` → Vega-Lite `facet` operator with `columns` parameter (wrapping layout)
- `facet.resolve` → `resolve: { scale: { ... } }` at the faceted spec level
- Both `row` and `column` → grid of panels (Vega-Lite supports this natively)

**Why this is better than overloading `rows`/`cols`:** In Tableau, putting `[Region]` and `[Category]` both on the Rows shelf creates nested row headers — but the intent (partitioning) is conflated with the axis assignment. In this DSL, if you want bars of revenue by country, faceted by region, you write:

```yaml
cols: country
rows: revenue
marks: bar
facet:
  row: region
```

This reads clearly: x-axis is country, y-axis is revenue, partitioned into rows by region.

#### `axis` — Axis Configuration

```yaml
axis:
  x:
    title: "Week"
    format: "%b %d"                # Date format
    grid: false
  y:
    title: "Revenue ($)"
    format: "$,.0f"
```

### 1.4 Encoding Inheritance

Encoding properties cascade through three levels. More specific always wins.

```
Top-level (marks, color, detail, size, opacity)
  └── applies to all measure entries that don't specify their own
        └── applies to all layer entries that don't specify their own
```

```yaml
marks: line                # Level 0: default for everything
color: country             # Level 0: default for everything

rows:
  - measure: revenue
    mark: bar              # Level 1: overrides line → bar for this entry
    # color: country       # Inherited from Level 0
    layer:
      - measure: arpu
        mark:              # Level 2: overrides for this layer only
          type: line
          style: dashed
        color: "#666666"   # Level 2: overrides country → fixed grey
```

Resulting marks: revenue = bar, arpu = dashed line.
Resulting colors: revenue = country (inherited), arpu = #666666 (override).

**`detail: null` is a valid override.** When a layer entry explicitly sets `detail: null`, it does NOT inherit the parent's detail — it aggregates across that dimension. This is critical for patterns like "bars by country + overall average line."

### 1.5 Complete Examples

#### Example 1: Simple Bar Chart

```yaml
sheet: "Revenue by Country"
data:
  model: orders
  measures: [revenue]
  dimensions: [country]

cols: country
rows: revenue
marks: bar
color: country
sort:
  field: revenue
  order: descending
tooltip: [country, revenue]
```

**Translates to Vega-Lite:**
```json
{
  "$schema": "https://vega.github.io/schema/vega-lite/v6.json",
  "mark": "bar",
  "encoding": {
    "x": { "field": "country", "type": "nominal", "sort": { "field": "revenue", "order": "descending" } },
    "y": { "field": "revenue", "type": "quantitative" },
    "color": { "field": "country", "type": "nominal" },
    "tooltip": [
      { "field": "country", "type": "nominal" },
      { "field": "revenue", "type": "quantitative" }
    ]
  }
}
```

#### Example 2: Stacked Panels (Same Mark)

```yaml
sheet: "Key Metrics"
data:
  model: orders
  measures: [revenue, orders, arpu]
  dimensions: [week]
  time_grain: { field: week, grain: week }

cols: week
marks: line

rows:
  - measure: revenue
  - measure: orders
  - measure: arpu

tooltip: [week]
```

Three stacked panels sharing the x-axis, each rendering one measure as a line chart. All entries share the same mark, so the translator uses Vega-Lite `repeat`.

#### Example 3: Stacked Panels (Different Marks)

```yaml
sheet: "Revenue and Orders"
data:
  model: orders
  measures: [revenue, orders]
  dimensions: [week, country]
  time_grain: { field: week, grain: week }

cols: week
rows:
  - measure: revenue
    mark: bar
    color: country
  - measure: orders
    mark: line
    color: country

tooltip: [week, country]
```

Two stacked panels with different marks — the translator uses Vega-Lite `vconcat`.

#### Example 4: Dual Axis — Bars + Dashed Line (Layers)

```yaml
sheet: "Revenue & ARPU by Week"
data:
  model: orders
  measures: [revenue, arpu]
  dimensions: [country, week]
  time_grain: { field: week, grain: week }

cols: week
rows:
  - measure: revenue
    mark: bar
    color: country
    detail: country
    layer:
      - measure: arpu
        mark:
          type: line
          style: dashed
        color: "#666666"
    axis: independent

filters:
  - field: date
    operator: between
    range: ["2024-01-01", "2024-12-31"]

tooltip: [week, country, revenue, arpu]
```

**Translates to Vega-Lite:**
```json
{
  "$schema": "https://vega.github.io/schema/vega-lite/v6.json",
  "layer": [
    {
      "mark": "bar",
      "encoding": {
        "x": { "field": "week", "type": "temporal" },
        "y": { "field": "revenue", "type": "quantitative", "title": "Revenue" },
        "color": { "field": "country", "type": "nominal" },
        "detail": { "field": "country", "type": "nominal" },
        "tooltip": [
          { "field": "week", "type": "temporal" },
          { "field": "country", "type": "nominal" },
          { "field": "revenue", "type": "quantitative" }
        ]
      }
    },
    {
      "mark": { "type": "line", "strokeDash": [6, 4] },
      "encoding": {
        "x": { "field": "week", "type": "temporal" },
        "y": { "field": "arpu", "type": "quantitative", "title": "ARPU" },
        "color": { "value": "#666666" },
        "tooltip": [
          { "field": "week", "type": "temporal" },
          { "field": "arpu", "type": "quantitative" }
        ]
      }
    }
  ],
  "resolve": { "scale": { "y": "independent" } },
  "transform": [
    { "filter": { "and": [
      { "field": "date", "gte": "2024-01-01" },
      { "field": "date", "lte": "2024-12-31" }
    ]}}
  ]
}
```

#### Example 5: KPI / Big Number Sheet

```yaml
sheet: "Total Revenue"
data:
  model: orders
  measures: [revenue]
  dimensions: []

marks: text
kpi:
  measure: revenue
  format: "$,.0f"
  comparison:
    measure: revenue_prior_period
    format: "+.1%"
    type: percent_change
```

KPI sheets are a special case. The translator renders these as styled text blocks rather than traditional charts. The `kpi` property triggers a specific template that displays the big number, optional comparison value, and optional sparkline.

#### Example 6: Faceted Bar Chart (Small Multiples by Region)

```yaml
sheet: "Revenue by Country, by Region"
data:
  model: orders
  measures: [revenue]
  dimensions: [country, region]

cols: country
rows: revenue
marks: bar
color: country
sort:
  field: revenue
  order: descending
tooltip: [country, region, revenue]

facet:
  row: region
  resolve:
    scale:
      y: independent
```

Note how `cols` and `rows` define the axes within each panel, while `facet.row` controls the partitioning. This separation is unambiguous — there is no question about whether `region` is an axis or a partition dimension.

#### Example 7: Wrapping Facet (Country Grid)

```yaml
sheet: "Revenue Trend by Country"
data:
  model: orders
  measures: [revenue]
  dimensions: [country, month]
  time_grain:
    field: month
    grain: month

cols: month
rows: revenue
marks: line
tooltip: [month, country, revenue]

facet:
  field: country
  columns: 4
  sort: descending
```

This creates a 4-column wrapping grid where each panel shows the revenue trend line for one country.

#### Example 8: Stacked Layers (Panels Where One Has Layers)

```yaml
sheet: "Revenue+ARPU Panel, Orders Panel"
data:
  model: orders
  measures: [revenue, arpu, orders]
  dimensions: [week]
  time_grain: { field: week, grain: week }

cols: week
rows:
  - measure: revenue
    mark: bar
    layer:
      - measure: arpu
        mark:
          type: line
          style: dashed
        color: "#666666"
    axis: independent

  - measure: orders
    mark: line

tooltip: [week, revenue, arpu, orders]
```

Two stacked panels. The first has revenue bars with an ARPU line layered on top (independent axes). The second is just orders as a line. The translator uses `vconcat` where the first child is a `layer` spec and the second is a simple spec.

#### Example 9: Revenue vs Target vs Forecast (Shared Axis)

```yaml
sheet: "Revenue vs Plan"
data:
  model: orders
  measures: [revenue, target, forecast]
  dimensions: [month]
  time_grain: { field: month, grain: month }

cols: month
rows:
  - measure: revenue
    mark: line
    color: "#4A90D9"
    layer:
      - measure: target
        mark:
          type: line
          style: dashed
        color: "#cccccc"
      - measure: forecast
        mark:
          type: line
          style: dotted
        color: "#E5A84B"
    # No axis: independent → shared y-axis (all are $ values)

tooltip: [month, revenue, target, forecast]
```

#### Example 10: Bars by Country + Overall Average Line

```yaml
sheet: "Revenue by Country + Average"
data:
  model: orders
  measures: [revenue, avg_revenue]
  dimensions: [country]

cols: country
rows:
  - measure: revenue
    mark: bar
    color: country
    layer:
      - measure: avg_revenue
        mark: rule
        color: "#666666"
        detail: null

sort:
  field: revenue
  order: descending
tooltip: [country, revenue, avg_revenue]
```

Note `detail: null` on the layer — this prevents the average line from inheriting the parent's color-by-country disaggregation.

#### Example 11: Layers + Facet

```yaml
sheet: "Revenue & ARPU by Week, by Region"
data:
  model: orders
  measures: [revenue, arpu]
  dimensions: [region, week]
  time_grain: { field: week, grain: week }

cols: week
rows:
  - measure: revenue
    mark: bar
    layer:
      - measure: arpu
        mark:
          type: line
          style: dashed
        color: "#E5A84B"
    axis: independent

facet:
  row: region

tooltip: [week, region, revenue, arpu]
```

Each region panel contains both the bar and line layers. `facet.py` wraps the layer spec identically to how it wraps a simple spec.

#### Example 12: Scatter Variant — Measure on Cols, Stacked on Rows

```yaml
sheet: "Sales vs Multiple Metrics"
data:
  model: orders
  measures: [sales, revenue, orders, arpu]
  dimensions: [country]

cols: sales
marks: circle
color: country

rows:
  - measure: revenue
  - measure: orders
  - measure: arpu

tooltip: [country, sales]
```

Three scatter panels stacked vertically: sales vs revenue, sales vs orders, sales vs arpu. Everything works symmetrically — if `cols` has the multi-measure list, panels stack horizontally.

## Part 2: LLM Integration

### 2.1 LLM's Role

The LLM translates natural language into the Chart DSL. It does not generate Vega-Lite, CSS, SQL, or any other code. Its output is validated against the DSL schema before proceeding.

### 2.2 Prompt → Chart DSL

**User:** "Show me weekly revenue by country as a bar chart with an ARPU trend line"

**LLM outputs:**
```yaml
sheet: "Revenue & ARPU by Week"
data:
  model: orders
  measures: [revenue, arpu]
  dimensions: [country, week]
  time_grain:
    field: week
    grain: week

cols: week
rows:
  - measure: revenue
    mark: bar
    color: country
    detail: country
    layer:
      - measure: arpu
        mark:
          type: line
          style: dashed
        color: "#666666"
    axis: independent

tooltip: [week, country, revenue, arpu]
```

### 2.3 Validation Layer

Before the DSL is passed to the translator, it must be validated:

1. **Schema validation:** Does the YAML conform to the DSL grammar? Are required fields present? Are values within allowed enums?
2. **Semantic layer validation:** Do referenced metrics and dimensions exist in the semantic layer? Is the model name valid?
3. **Consistency checks:** At most one multi-measure shelf? All referenced fields in `data.measures` or `data.dimensions`?
4. **If validation fails:** Return structured error to the LLM with the specific failure, and prompt for a corrected output.

## Part 3: Supported Chart Patterns

The DSL-to-Vega-Lite translator handles a finite set of chart patterns. Each pattern is a "recipe" — a known DSL structure that maps to a specific Vega-Lite composition. New patterns can be added over time.

### 3.1 Pattern Catalog

| Pattern | DSL Trigger | Vega-Lite Structure |
|---|---|---|
| Simple bar | `marks: bar`, single measure on rows | Single `mark: bar` spec |
| Grouped bar | `marks: bar`, `color: dimension` | `mark: bar` with `color` and `xOffset` encoding |
| Stacked bar | `marks: bar`, `color: dimension`, `stack: true` | `mark: bar` with `color`, default stacking |
| Line chart | `marks: line`, temporal on cols | Single `mark: line` spec |
| Multi-line | `marks: line`, `color: dimension` | `mark: line` with `color` encoding |
| Area chart | `marks: area` | Single `mark: area` spec |
| Stacked area | `marks: area`, `color: dimension` | `mark: area` with `color`, default stacking |
| Scatter plot | `marks: circle`, measures on both axes | `mark: circle`, both `x` and `y` quantitative |
| Stacked panels (same mark) | `rows` is list, all same mark | `repeat` with `row` |
| Stacked panels (mixed marks) | `rows` is list, different marks | `vconcat` array |
| Layered multi-axis | Entry with `layer` list | `layer` array with `resolve` |
| Stacked layers | `rows` is list, some with `layer` | `vconcat` of layer + simple specs |
| KPI / Big number | `marks: text` + `kpi` property | Custom text template |
| Facet rows | `facet.row` present | `facet` operator with `row` channel |
| Facet columns | `facet.column` present | `facet` operator with `column` channel |
| Facet grid | `facet.row` + `facet.column` | `facet` operator with both `row` and `column` |
| Facet wrap | `facet.field` + `facet.columns` | `facet` operator with `columns` (wrapping) |
| Histogram | `marks: bar`, single measure on both axes with `bin: true` | `mark: bar` with `bin` on `x` |
| Heatmap | `marks: rect`, two dimensions + measure on color | `mark: rect` with `color: quantitative` |
| Pie/Donut | `marks: arc` | `mark: arc` with `theta` encoding |
| Map | `marks: geoshape` | `mark: geoshape` with projection |

Any of the above base patterns can be combined with any facet pattern. Stacked panels and layered specs can be faceted — `facet.py` is inner-spec-agnostic.

### 3.2 Adding New Patterns

To add a new chart pattern:
1. Define the DSL trigger conditions (which property combinations identify this pattern)
2. Write the Vega-Lite template that the pattern compiles to
3. Add example DSL → Vega-Lite pairs to the test suite
4. Add examples to the LLM system prompt so it can generate the pattern from natural language

## Part 4: Pydantic Schema

```python
class LayerEntry(BaseModel):
    """A measure layered on top of a parent row/col entry."""
    measure: str
    mark: MarkSpec | None = None
    color: ColorSpec | None = None
    detail: str | None = None
    size: str | int | float | None = None
    opacity: float | None = Field(None, ge=0.0, le=1.0)


class MeasureEntry(BaseModel):
    """One entry on the multi-measure shelf (row or col)."""
    measure: str
    mark: MarkSpec | None = None
    color: ColorSpec | None = None
    detail: str | None = None
    size: str | int | float | None = None
    opacity: float | None = Field(None, ge=0.0, le=1.0)
    axis: Literal["independent", "shared"] | None = None
    layer: list[LayerEntry] | None = None


# A shelf is either a single field name or a list of measure entries
ShelfSpec = Union[str, list[MeasureEntry]]


class ChartSpec(BaseModel):
    # ...
    cols: ShelfSpec | None = None
    rows: ShelfSpec | None = None
    # ...

    @model_validator(mode="after")
    def at_most_one_multi_measure_shelf(self):
        """Only one of rows/cols can be a multi-measure list."""
        rows_multi = isinstance(self.rows, list)
        cols_multi = isinstance(self.cols, list)
        if rows_multi and cols_multi:
            raise ValueError(
                "Only one of rows/cols can have multiple measures. "
                "Use the single-field shelf for the other axis."
            )
        return self
```

## Part 5: Translation Logic

```python
def translate_chart(spec: ChartSpec) -> dict:
    # Determine which shelf is multi-measure (if any)
    rows_is_multi = isinstance(spec.rows, list)
    cols_is_multi = isinstance(spec.cols, list)

    if not rows_is_multi and not cols_is_multi:
        # Phase 1 path: single mark chart
        return compile_single(spec)

    if rows_is_multi:
        return compile_multi_rows(spec)

    if cols_is_multi:
        return compile_multi_cols(spec)


def compile_multi_rows(spec: ChartSpec) -> dict:
    entries = spec.rows  # list[MeasureEntry]
    shared_x = build_x_encoding(spec.cols, resolver)

    if len(entries) == 1 and entries[0].layer:
        # Single entry with layers → pure layer spec (no concat)
        return compile_layer_entry(entries[0], shared_x, spec)

    if all(e.layer is None for e in entries):
        # All standalone → repeat or concat
        return compile_stacked_entries(entries, shared_x, spec)

    # Mix of standalone and layered → vconcat
    panels = []
    for entry in entries:
        if entry.layer:
            panels.append(compile_layer_entry(entry, shared_x, spec))
        else:
            panels.append(compile_single_entry(entry, shared_x, spec))

    return {"vconcat": panels}
```
