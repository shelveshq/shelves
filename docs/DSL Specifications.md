# DSL Specification: Chart Grammar & Layout Grammar

This document defines the two domain-specific languages (DSLs) that form the core authoring interface of the platform: the **Chart DSL** for defining individual visualizations, and the **Layout DSL** for composing dashboards.

Both DSLs are written in YAML. They are designed to be authored directly by BI analysts or generated by an LLM from natural language prompts. Both translate deterministically to their target formats — Chart DSL to Vega-Lite JSON, Layout DSL to CSS flexbox HTML.

---

## Part 1: Chart DSL

### 1.1 Design Philosophy

The Chart DSL uses Tableau's shelf mental model as its vocabulary. An analyst who thinks in terms of rows, columns, marks, color, and detail can write a chart spec without learning Vega-Lite. The DSL acts as a constrained intermediate representation — compact enough for an LLM to produce reliably, expressive enough to cover the most common BI chart patterns.

**Principles:**
- Vocabulary mirrors Tableau: `rows`, `cols`, `marks`, `color`, `detail`, `filters`, `tooltip`, `sort`
- **`rows`/`cols` are exclusively for axis encoding; `facet` is a separate property for chart partitioning.** This is a deliberate improvement over Tableau, where the Rows/Columns shelves conflate axis assignment with small-multiple generation. In this DSL, the intent is always unambiguous.
- References semantic layer metrics/dimensions by name (never raw SQL or column expressions)
- Styling is excluded — the theme layer handles all visual aesthetics
- Each YAML file defines exactly one chart ("sheet" in Tableau terms)

### 1.2 Grammar Reference

#### Top-Level Properties

```yaml
# chart_name.yaml
sheet: "Revenue by Country"           # Display name
description: "Weekly revenue bars..."  # Optional description

# Data source — references a semantic layer model
data:
  model: "orders"                      # Semantic layer model/cube name
  measures: [revenue, arpu]            # Metrics to query
  dimensions: [country, week]          # Dimensions to include

# Shelf assignments (axis encoding only — never used for partitioning)
cols: <shelf_definition>               # X-axis / horizontal encoding
rows: <shelf_definition>               # Y-axis / vertical encoding

# Mark definitions
marks: <mark_definition>               # Single mark or list (for dual axis)

# Encoding channels
color: <encoding_definition>           # Color mapping
detail: <encoding_definition>          # Disaggregation without visual encoding
size: <encoding_definition>            # Size mapping
tooltip: <tooltip_definition>          # Tooltip fields

# Interactions
filters: <filter_definition>           # Shelf filters (hardcoded in spec)
sort: <sort_definition>                # Sort order

# Multi-axis
axis: <axis_definition>                # Axis configuration (sync, independent)

# Partitioning (small multiples — separate from axis encoding)
facet: <facet_definition>              # Row, column, or wrapping grid partitioning
```

### 1.3 Property Definitions

#### `data` — Data Source

```yaml
data:
  model: "orders"                   # Required: semantic layer model name
  measures:                         # Required: list of metrics to query
    - revenue
    - arpu
  dimensions:                       # Required: list of dimensions
    - country
    - week
  time_grain:                       # Optional: time dimension granularity
    field: week
    grain: week                     # day | week | month | quarter | year
```

The translator uses `model`, `measures`, and `dimensions` to construct the semantic layer API call. The returned data is bound to the Vega-Lite spec's `data` property.

#### `cols` and `rows` — Axis Encoding (Not Partitioning)

`cols` defines what goes on the x-axis. `rows` defines what goes on the y-axis. These are strictly for axis encoding — they never partition the chart into small multiples. Use `facet` for partitioning.

```yaml
# Simple: single field
cols: week
rows: revenue

# Dual axis: multiple measures on the same axis
rows:
  dual_axis: true
  measures:
    - revenue
    - arpu
  axis: independent               # independent | shared (default: shared)
```

**Translation rules:**
- Single dimension on `cols` → `encoding.x` with `type: temporal/nominal/ordinal`
- Single measure on `rows` → `encoding.y` with `type: quantitative`
- `dual_axis: true` → `layer` spec with separate `y` encodings per measure
- `axis: independent` → `resolve: { scale: { y: "independent" } }`

**What NOT to do:** Do not put multiple dimensions on `rows` or `cols` to create nested headers or small multiples. Use the `facet` property instead. This keeps the DSL unambiguous — `rows`/`cols` always mean axis encoding.

#### `marks` — Mark Definitions

For single-measure charts, marks is a single type. For dual-axis charts, marks is a list corresponding to each measure.

```yaml
# Single mark
marks: bar

# Single mark with properties
marks:
  type: line
  style: dashed                    # solid (default) | dashed | dotted

# Dual axis — one mark per measure (order matches rows.measures)
marks:
  - type: bar
  - type: line
    style: dashed

# Mark with additional properties
marks:
  - type: bar
    opacity: 0.8
  - type: line
    style: dashed
    point: true                    # Show data points on line
```

**Supported mark types:**
| Mark | Vega-Lite `mark.type` | Notes |
|---|---|---|
| `bar` | `bar` | |
| `line` | `line` | `style: dashed` → `strokeDash: [6, 4]` |
| `area` | `area` | |
| `circle` | `circle` | For scatter plots |
| `square` | `square` | |
| `text` | `text` | Typically layered on another mark |
| `point` | `point` | Generic point mark |
| `rule` | `rule` | Reference lines |
| `tick` | `tick` | |
| `geoshape` | `geoshape` | Maps |

#### `color` — Color Encoding

Color can be a field mapping, a fixed value, or a per-mark definition for dual axis charts.

```yaml
# Map a dimension to color (categorical)
color: country

# Map a measure to color (sequential/gradient)
color:
  field: revenue
  type: quantitative

# Fixed color value (overrides theme)
color: "#4A90D9"

# Per-mark color (dual axis — order matches marks list)
color:
  - country                        # First mark: colored by country
  - "#666666"                      # Second mark: fixed dark grey
```

**Translation rules:**
- Field name → `encoding.color: { field: "country", type: "nominal" }`
- Hex value → `mark.color: "#4A90D9"` or `encoding.color: { value: "#666666" }`
- Per-mark list → each `layer[n]` gets its own `color` encoding

#### `detail` — Detail / Level of Disaggregation

Detail breaks marks into groups without applying a visual encoding. This is identical to Tableau's Detail shelf.

```yaml
# Single field
detail: country

# Per-mark detail (dual axis)
detail:
  - country                        # First mark: disaggregated by country
  - null                           # Second mark: aggregated (no disaggregation)
```

**Translation:** `detail: country` → `encoding.detail: { field: "country", type: "nominal" }`

A `null` value means the mark aggregates across all values of that dimension — critical for patterns like "bars by country + overall average line."

#### `size` — Size Encoding

```yaml
# Map field to size
size: revenue

# Fixed size
size: 200
```

#### `tooltip` — Tooltip Fields

```yaml
# Simple list
tooltip: [country, revenue, arpu]

# With formatting
tooltip:
  - field: country
  - field: revenue
    format: "$,.0f"
  - field: arpu
    format: "$,.2f"
```

#### `filters` — Shelf Filters (Hardcoded)

These are data filters baked into the chart spec, not interactive filter panes (those live in the layout DSL).

```yaml
filters:
  - field: country
    operator: in                   # in | not_in | eq | neq | gt | lt | gte | lte | between
    values: ["US", "UK", "FR"]

  - field: revenue
    operator: gt
    value: 1000

  - field: date
    operator: between
    range: ["2024-01-01", "2024-12-31"]
```

**Translation:** Each filter becomes a `transform: [{ filter: ... }]` entry in the Vega-Lite spec.

#### `sort` — Sort Order

```yaml
# Sort by field value (ascending)
sort:
  field: revenue
  order: descending                # ascending | descending

# Explicit custom order
sort:
  field: country
  order: ["US", "UK", "FR", "DE", "JP"]

# Sort by encoding (e.g., sort x-axis bars by y value)
sort:
  encoding: y
  order: descending
```

#### `facet` — Chart Partitioning (Small Multiples)

Facet is the sole mechanism for partitioning a chart into small multiples. It is deliberately separate from `rows`/`cols` to maintain a clean distinction: `rows`/`cols` control what goes on each axis, `facet` controls how the chart is split into panels.

```yaml
# Facet into rows (vertical stack of panels)
facet:
  row: region

# Facet into columns (horizontal row of panels)
facet:
  column: product_category

# Facet into both rows and columns (grid)
facet:
  row: region
  column: product_category

# Wrapping facet (single dimension, wraps into a grid)
facet:
  field: country
  columns: 4                       # Wrap after 4 columns
  sort: ascending                  # Optional sort order for panels

# Facet with independent scales (each panel gets its own axis range)
facet:
  row: region
  resolve:
    scale:
      y: independent               # independent | shared (default: shared)
```

**Translation rules:**
- `facet.row` → Vega-Lite `row` encoding channel or `facet` operator with `row`
- `facet.column` → Vega-Lite `column` encoding channel or `facet` operator with `column`
- `facet.field` + `facet.columns` → Vega-Lite `facet` operator with `columns` parameter (wrapping layout)
- `facet.resolve` → `resolve: { scale: { ... } }` at the faceted spec level
- Both `row` and `column` → grid of panels (Vega-Lite supports this natively)

**Why this is better than overloading `rows`/`cols`:** In Tableau, putting `[Region]` and `[Category]` both on the Rows shelf creates nested row headers — but the intent (partitioning) is conflated with the axis assignment. In this DSL, if you want bars of revenue by country, faceted by region, you write:

```yaml
cols: country
rows: revenue
marks: bar
facet:
  row: region
```

This reads clearly: x-axis is country, y-axis is revenue, partitioned into rows by region.

#### `axis` — Axis Configuration

```yaml
axis:
  x:
    title: "Week"
    format: "%b %d"                # Date format
    grid: false
  y:
    title: "Revenue ($)"
    format: "$,.0f"
```

### 1.4 Complete Examples

#### Example 1: Simple Bar Chart

```yaml
sheet: "Revenue by Country"
data:
  model: orders
  measures: [revenue]
  dimensions: [country]

cols: country
rows: revenue
marks: bar
color: country
sort:
  field: revenue
  order: descending
tooltip: [country, revenue]
```

**Translates to Vega-Lite:**
```json
{
  "$schema": "https://vega.github.io/schema/vega-lite/v6.json",
  "mark": "bar",
  "encoding": {
    "x": { "field": "country", "type": "nominal", "sort": { "field": "revenue", "order": "descending" } },
    "y": { "field": "revenue", "type": "quantitative" },
    "color": { "field": "country", "type": "nominal" },
    "tooltip": [
      { "field": "country", "type": "nominal" },
      { "field": "revenue", "type": "quantitative" }
    ]
  }
}
```

#### Example 2: Dual Axis — Bars + Dashed Line

```yaml
sheet: "Revenue & ARPU by Week"
data:
  model: orders
  measures: [revenue, arpu]
  dimensions: [country, week]
  time_grain:
    field: week
    grain: week

cols: week
rows:
  dual_axis: true
  measures: [revenue, arpu]
  axis: independent

marks:
  - type: bar
  - type: line
    style: dashed

color:
  - country
  - "#666666"

detail:
  - country
  - null

filters:
  - field: date
    operator: between
    range: ["2024-01-01", "2024-12-31"]

tooltip: [week, country, revenue, arpu]
```

**Translates to Vega-Lite:**
```json
{
  "$schema": "https://vega.github.io/schema/vega-lite/v6.json",
  "layer": [
    {
      "mark": "bar",
      "encoding": {
        "x": { "field": "week", "type": "temporal" },
        "y": { "field": "revenue", "type": "quantitative", "title": "Revenue" },
        "color": { "field": "country", "type": "nominal" },
        "detail": { "field": "country", "type": "nominal" },
        "tooltip": [
          { "field": "week", "type": "temporal" },
          { "field": "country", "type": "nominal" },
          { "field": "revenue", "type": "quantitative" }
        ]
      }
    },
    {
      "mark": { "type": "line", "strokeDash": [6, 4] },
      "encoding": {
        "x": { "field": "week", "type": "temporal" },
        "y": { "field": "arpu", "type": "quantitative", "title": "ARPU" },
        "color": { "value": "#666666" },
        "tooltip": [
          { "field": "week", "type": "temporal" },
          { "field": "arpu", "type": "quantitative" }
        ]
      }
    }
  ],
  "resolve": { "scale": { "y": "independent" } },
  "transform": [
    { "filter": { "and": [
      { "field": "date", "gte": "2024-01-01" },
      { "field": "date", "lte": "2024-12-31" }
    ]}}
  ]
}
```

#### Example 3: KPI / Big Number Sheet

```yaml
sheet: "Total Revenue"
data:
  model: orders
  measures: [revenue]
  dimensions: []

marks: text
kpi:
  measure: revenue
  format: "$,.0f"
  comparison:
    measure: revenue_prior_period
    format: "+.1%"
    type: percent_change
```

KPI sheets are a special case. The translator renders these as styled text blocks rather than traditional charts. The `kpi` property triggers a specific template that displays the big number, optional comparison value, and optional sparkline.

#### Example 4: Faceted Bar Chart (Small Multiples by Region)

```yaml
sheet: "Revenue by Country, by Region"
data:
  model: orders
  measures: [revenue]
  dimensions: [country, region]

cols: country
rows: revenue
marks: bar
color: country
sort:
  field: revenue
  order: descending
tooltip: [country, region, revenue]

facet:
  row: region
  resolve:
    scale:
      y: independent
```

**Translates to Vega-Lite:**
```json
{
  "$schema": "https://vega.github.io/schema/vega-lite/v6.json",
  "facet": { "row": { "field": "region", "type": "nominal" } },
  "spec": {
    "mark": "bar",
    "encoding": {
      "x": { "field": "country", "type": "nominal", "sort": { "field": "revenue", "order": "descending" } },
      "y": { "field": "revenue", "type": "quantitative" },
      "color": { "field": "country", "type": "nominal" },
      "tooltip": [
        { "field": "country", "type": "nominal" },
        { "field": "region", "type": "nominal" },
        { "field": "revenue", "type": "quantitative" }
      ]
    }
  },
  "resolve": { "scale": { "y": "independent" } }
}
```

Note how `cols` and `rows` define the axes within each panel, while `facet.row` controls the partitioning. This separation is unambiguous — there is no question about whether `region` is an axis or a partition dimension.

#### Example 5: Wrapping Facet (Country Grid)

```yaml
sheet: "Revenue Trend by Country"
data:
  model: orders
  measures: [revenue]
  dimensions: [country, month]
  time_grain:
    field: month
    grain: month

cols: month
rows: revenue
marks: line
tooltip: [month, country, revenue]

facet:
  field: country
  columns: 4
  sort: descending
```

This creates a 4-column wrapping grid where each panel shows the revenue trend line for one country. Panels are sorted by revenue (descending), so the highest-revenue countries appear first.

#### Example 6: Scatter Plot with Trend Line

```yaml
sheet: "Revenue vs ARPU Scatter"
data:
  model: orders
  measures: [revenue, arpu]
  dimensions: [country]

cols: revenue
rows: arpu
marks: circle
color: country
size: order_count
tooltip: [country, revenue, arpu, order_count]

layers:
  - type: trend_line
    method: linear                 # linear | log | poly | exp
```

## Part 2: Layout DSL

### 2.1 Design Philosophy

The Layout DSL describes dashboard structure as nested horizontal and vertical containers — the same model Tableau uses for dashboard layouts. Every element in the layout tree is one of four types: container, sheet (chart reference), filter pane, or text block.

**Principles:**
- Fixed dashboard canvas size (e.g., 1440×900px) — eliminates responsive design complexity
- Horizontal and vertical containers nest arbitrarily deep
- Every element has explicit or inherited dimensions
- Filter panes are first-class layout components with targeting rules
- Pure structural description — no styling (theme handles that), no data logic

### 2.2 Grammar Reference

#### Dashboard Root

```yaml
# dashboard_name.yaml
dashboard: "Sales Overview"
description: "Weekly sales performance dashboard"
canvas:
  width: 1440
  height: 900

# Component definitions
components:
  <component_name>: <component_definition>
  ...

# Layout tree
layout:
  <tree_definition>
```

#### Component Types

##### Container

```yaml
kpis_row:
  type: container
  orientation: horizontal          # horizontal | vertical
  height: 200px                    # Optional fixed height
  width: 100%                      # Optional fixed width (default: fills parent)
  gap: 16px                        # Optional gap between children
  padding: 16px                    # Optional internal padding
```

##### Sheet (Chart Reference)

```yaml
revenue_chart:
  type: sheet
  spec: "charts/revenue_by_country.yaml"   # Path to chart DSL file
  data_source: "orders"                     # Semantic layer model (can override chart default)
  width: 50%                               # Relative or fixed width within parent
  height: 100%                             # Relative or fixed height within parent
  min_width: 300px                         # Optional minimum
```

##### Filter Pane

```yaml
country_filter:
  type: filter
  field: country                   # Dimension name from semantic layer
  input: dropdown                  # dropdown | multiselect | slider | range | radio | checkbox | date_range
  label: "Country"                 # Display label
  default: null                    # Default selection (null = all)
  targets: all                     # all | [list of sheet names]
  width: 200px
```

**Filter input types:**

| Input Type | Vega-Lite `params` Binding | Use Case |
|---|---|---|
| `dropdown` | `bind: { input: "select" }` | Single selection from list |
| `multiselect` | `bind: { input: "select", multiple: true }` | Multiple selections |
| `slider` | `bind: { input: "range" }` | Numeric range (single value) |
| `range` | `bind: { input: "range" }` (two-ended) | Numeric range (min/max) |
| `radio` | `bind: { input: "radio" }` | Single selection, all visible |
| `checkbox` | `bind: { input: "checkbox" }` | Toggle on/off |
| `date_range` | Custom component | Start/end date picker |

**Filter targeting:** The `targets` property controls which charts respond to the filter.
- `all` — every sheet in the dashboard using the same data model
- `[revenue_chart, kpi_1]` — only the named sheets

##### Text Block

```yaml
dashboard_title:
  type: text
  content: "Sales Performance Dashboard"
  style: title                     # title | subtitle | body | caption
  align: left                      # left | center | right
  height: 40px
```

##### Spacer / Blank

```yaml
spacer_1:
  type: blank
  width: 16px
  height: 16px
```

### 2.3 Layout Tree

The layout tree defines the nesting structure. It references components defined in the `components` section and arranges them within containers.

```yaml
layout:
  root:
    type: container
    orientation: vertical
    children:
      - ref: title_bar
        children:
          - ref: dashboard_title
          - ref: date_filter
          - ref: country_filter

      - ref: kpis_row
        children:
          - ref: kpi_revenue
          - ref: kpi_orders
          - ref: kpi_arpu
          - ref: kpi_customers

      - ref: charts_row
        children:
          - ref: revenue_chart
          - ref: orders_chart
```

**Translation to CSS flexbox:**
- `type: container, orientation: horizontal` → `display: flex; flex-direction: row;`
- `type: container, orientation: vertical` → `display: flex; flex-direction: column;`
- `height: 200px` → `height: 200px;`
- `width: 50%` → `flex: 0 0 50%;`
- `gap: 16px` → `gap: 16px;`
- Sheet components become `<div>` wrappers with a Vega-Lite embed call inside

### 2.4 Complete Dashboard Example

          - ref: revenue_trend

### 2.5 Filter Propagation Model

When a user interacts with a filter pane, the system must propagate the selection to targeted charts. The mechanism depends on the rendering approach:

**Option A: Vega-Lite native params (for charts within a single Vega spec)**
- All charts in the dashboard are composed into a single `vconcat`/`hconcat` Vega-Lite spec
- Filters are `params` defined at the top level
- Cross-filtering happens natively via shared parameters
- Simpler implementation but limits independence of chart specs

**Option B: Application-level state management (recommended for flexibility)**
- Each chart is an independent Vega-Lite embed
- Filter selections are captured in application state (React state, etc.)
- When a filter changes, the application re-queries the semantic layer with updated filters and pushes new data to affected charts via the Vega View API
- More flexible, supports targeting specific charts, and works with VegaFusion

The Layout DSL's `targets` property on filter components drives the propagation:
- `targets: all` → state change triggers re-query for all sheets
- `targets: [revenue_chart]` → only the named sheet re-queries

## Part 3: LLM Integration

### 3.1 LLM's Role

The LLM translates natural language into the Chart DSL and/or Layout DSL. It does not generate Vega-Lite, CSS, SQL, or any other code. Its output is validated against the DSL schema before proceeding.

### 3.2 Prompt → Chart DSL

**User:** "Show me weekly revenue by country as a bar chart with an ARPU trend line"

**LLM outputs:**
```yaml
sheet: "Revenue & ARPU by Week"
data:
  model: orders
  measures: [revenue, arpu]
  dimensions: [country, week]
  time_grain:
    field: week
    grain: week

cols: week
rows:
  dual_axis: true
  measures: [revenue, arpu]
  axis: independent

marks:
  - type: bar
  - type: line
    style: dashed

color:
  - country
  - "#666666"

detail:
  - country
  - null

tooltip: [week, country, revenue, arpu]
```

### 3.3 Prompt → Layout DSL

**User:** "Create a dashboard with four KPIs at the top, two charts below, and country and date filters"

**LLM outputs:**
```yaml

    type: text
    content: "Overview Dashboard"
    style: title

  country_filter:
    type: filter
    field: country
    input: multiselect
    targets: all

  date_filter:
    type: filter
    field: date
    input: date_range
    targets: all

  header:
    type: container
    orientation: horizontal
    height: 60px

  kpi_row:
    type: container
    orientation: horizontal
    height: 150px

  chart_row:
    type: container
    orientation: horizontal
    height: 500px

  kpi_1: { type: sheet, spec: "charts/kpi_1.yaml", width: 25% }
  kpi_2: { type: sheet, spec: "charts/kpi_2.yaml", width: 25% }
  kpi_3: { type: sheet, spec: "charts/kpi_3.yaml", width: 25% }
  kpi_4: { type: sheet, spec: "charts/kpi_4.yaml", width: 25% }
  chart_1: { type: sheet, spec: "charts/chart_1.yaml", width: 50% }
  chart_2: { type: sheet, spec: "charts/chart_2.yaml", width: 50% }

  root:
    type: container
    orientation: vertical
    children:
      - ref: header
        children: [title, date_filter, country_filter]
      - ref: kpi_row
        children: [kpi_1, kpi_2, kpi_3, kpi_4]
      - ref: chart_row
        children: [chart_1, chart_2]
```

### 3.4 Validation Layer

Before the DSL is passed to the translator, it must be validated:

1. **Schema validation:** Does the YAML conform to the DSL grammar? Are required fields present? Are values within allowed enums?
2. **Semantic layer validation:** Do referenced metrics and dimensions exist in the semantic layer? Is the model name valid?
3. **Consistency checks:** Do `marks`, `color`, and `detail` lists have the same length as `rows.measures` for dual-axis charts?
4. **If validation fails:** Return structured error to the LLM with the specific failure, and prompt for a corrected output.

## Part 4: Supported Chart Patterns

The DSL-to-Vega-Lite translator must handle a finite set of chart patterns. Each pattern is a "recipe" — a known DSL structure that maps to a specific Vega-Lite composition. New patterns can be added over time.

### 4.1 Pattern Catalog

| Pattern | DSL Trigger | Vega-Lite Structure |
|---|---|---|
| Simple bar | `marks: bar`, single measure on rows | Single `mark: bar` spec |
| Grouped bar | `marks: bar`, `color: dimension` | `mark: bar` with `color` and `xOffset` encoding |
| Stacked bar | `marks: bar`, `color: dimension`, `stack: true` | `mark: bar` with `color`, default stacking |
| Line chart | `marks: line`, temporal on cols | Single `mark: line` spec |
| Multi-line | `marks: line`, `color: dimension` | `mark: line` with `color` encoding |
| Area chart | `marks: area` | Single `mark: area` spec |
| Stacked area | `marks: area`, `color: dimension` | `mark: area` with `color`, default stacking |
| Scatter plot | `marks: circle`, measures on both axes | `mark: circle`, both `x` and `y` quantitative |
| Dual axis | `rows.dual_axis: true` | `layer` with separate `y` encodings + `resolve` |
| KPI / Big number | `marks: text` + `kpi` property | Custom text template |
| Facet rows | `facet.row` present | `facet` operator with `row` channel |
| Facet columns | `facet.column` present | `facet` operator with `column` channel |
| Facet grid | `facet.row` + `facet.column` | `facet` operator with both `row` and `column` |
| Facet wrap | `facet.field` + `facet.columns` | `facet` operator with `columns` (wrapping) |
| Histogram | `marks: bar`, single measure on both axes with `bin: true` | `mark: bar` with `bin` on `x` |
| Heatmap | `marks: rect`, two dimensions + measure on color | `mark: rect` with `color: quantitative` |
| Pie/Donut | `marks: arc` | `mark: arc` with `theta` encoding |
| Treemap | Future | Requires Vega (not Vega-Lite) or custom transform |
| Map | `marks: geoshape` | `mark: geoshape` with projection |

Any of the above base patterns can be combined with any facet pattern. For example, a dual-axis chart with `facet.row: region` produces a layered spec wrapped in a facet operator — each region panel contains both the bar and line layers.

### 4.2 Adding New Patterns

To add a new chart pattern:
1. Define the DSL trigger conditions (which property combinations identify this pattern)
2. Write the Vega-Lite template that the pattern compiles to
3. Add example DSL → Vega-Lite pairs to the test suite
4. Add examples to the LLM system prompt so it can generate the pattern from natural language
